------------------------ Schema, Router, và Controller
Hãy tưởng tượng Schema, Router, và Controller là 3 lớp bảo vệ và xử lý dữ liệu từ ngoài vào trong:
Schema (Cổng kiểm soát): Kiểm tra xem dữ liệu khách hàng gửi lên có "đúng hình dạng" không (có đủ tên, mật khẩu không, có bị trống không).
Router (Lễ tân): Tiếp nhận yêu cầu (Request), lấy dữ liệu đã kiểm tra từ Schema, đưa cho Controller xử lý, rồi lấy kết quả trả về cho khách hàng.
Controller (Nhân viên xử lý): Chứa "não bộ" của ứng dụng. Nó thực hiện tính toán, truy vấn SQL Server, logic nghiệp vụ.
Bảng so sánh để bạn dễ hình dung:
Thành phần		Vai trò chính		Thao tác cụ thể
Schema (pydantic)	Định dạng dữ liệu	"Dữ liệu login phải có username (string) và password (string)."
Router (fastapi)	Định hướng (Routes)	"Nếu ai gọi /api/login bằng phương thức POST, hãy chuyển cho Controller."
Controller (python)	Nghiệp vụ (Logic)	"Vào SQL Server tìm xem có ai tên này không, mật khẩu có khớp không."
Luồng chạy thực tế sẽ như thế này:
Client (JS/Ajax) gửi một gói JSON lên /auth/api/login.
Schema nhảy vào kiểm tra:
Nếu JSON thiếu trường username -> Trả lỗi 422 ngay lập tức (Router và Controller chưa cần làm gì cả).
Nếu JSON hợp lệ -> Chuyển dữ liệu sang Router.
Router nhận dữ liệu sạch từ Schema, rồi gọi: AuthController.login(data.username, data.password).
Controller kết nối SQL Server, lấy kết quả trả về cho Router.
Router đóng gói kết quả đó (có thể kèm theo việc lưu Session) và gửi trả về cho Client.

------------------------------ Router + Schema = View (Flask)
Thực ra, Router + Schema tốt hơn rất nhiều khi dự án của bạn bắt đầu có hơn 5-10 chức năng. Nếu bạn chỉ làm 1-2 trang web nhỏ, dùng dict như cũ sẽ nhanh hơn, nhưng với một hệ thống HR (Nhân sự) có nhiều bảng dữ liệu, Schema mang lại 3 lợi ích "sống còn":
1. Tự động hóa việc kiểm tra lỗi (Validation)
Cách cũ (dict): Bạn phải viết if not data.get("username"): return "Thiếu tên". Nếu có 20 trường thông tin nhân viên, bạn sẽ phải viết 20 dòng if.
Cách mới (Schema): Bạn chỉ cần định nghĩa username: str. Nếu Client gửi thiếu hoặc gửi sai kiểu dữ liệu, FastAPI tự động chặn lại và trả lỗi 422 ngay lập tức. Controller của bạn không bao giờ phải xử lý dữ liệu "rác".
2. Tài liệu API "Xịn" tự động (Swagger UI)
Đây là điểm ăn tiền nhất của FastAPI. Khi bạn dùng Schema, bạn chỉ cần truy cập /docs, hệ thống sẽ tự tạo một trang web:
Liệt kê tất cả API.
Hiển thị chính xác Client cần gửi lên cái gì (JSON mẫu).
Cho phép bạn ấn nút "Try it out" để test trực tiếp mà không cần dùng Postman.
Nếu dùng dict, trang này sẽ trống không, bạn không biết API đó yêu cầu dữ liệu gì.

Trong permission_ctl.py, chúng ta vẫn dùng dict vì 2 lý do kỹ thuật của thư viện pyodbc:

Hiệu năng: Việc ép toàn bộ kết quả từ SQL vào một Class Schema ngay tại Controller sẽ làm chậm tốc độ xử lý nếu danh sách nhân viên lên đến hàng nghìn người.

Controller: Dùng dict để "nhào nặn" dữ liệu từ SQL Server cho nhanh và linh hoạt.

Để ghi log cho toàn bộ hệ thống một cách tự động mà không phải viết lại hàm write_user_log trong từng Router, cách tối ưu nhất trong FastAPI là sử dụng Middleware.
Middleware sẽ giống như một "trạm kiểm soát" ở cổng chính, nó tự động bắt lấy mọi Request (thêm, sửa, xóa) và ghi log trước khi trả kết quả về cho khách hàng.
1. Cập nhật database.py (Hàm ghi log hỗ trợ Middleware)
Chúng ta chỉnh lại một chút để hàm này nhận các thông tin thô từ Middleware.
python
def write_system_log(user_name: str, method: str, path: str, status_code: int, ip: str):
    """Ghi log tự động cho mọi request"""
    try:
        # Chỉ ghi log cho các thao tác thay đổi dữ liệu (POST, PUT, DELETE)
        # Hoặc ghi tất cả tùy bạn, ở đây tôi lọc để tránh rác DB
        if method in ["POST", "PUT", "DELETE"] or "api" in path:
            with pyodbc.connect(settings.connection_string) as conn:
                cursor = conn.cursor()
                sql = """
                    INSERT INTO UserLogs (UserID, Action, IPAddress, CreatedAt) 
                    VALUES (?, ?, ?, GETDATE())
                """
                action = f"{method} {path} - Status: {status_code}"
                cursor.execute(sql, (user_name, action, ip))
                conn.commit()
    except Exception as e:
        print(f"❌ Lỗi Middleware Log: {e}")

2. Cấu hình Middleware trong main.py
Bạn thêm đoạn code này vào file main.py, ngay sau phần cấu hình SessionMiddleware.
python
from fastapi import Request
from database import write_system_log
import time

@app.middleware("http")
async def log_requests(request: Request, call_next):
    # 1. Lấy thông tin trước khi xử lý Request
    user_name = request.session.get("user_name", "Anonymous")
    ip = request.client.host
    path = request.url.path
    method = request.method

    # 2. Cho phép Request đi tiếp vào Router/Controller
    response = await call_next(request)

    # 3. Sau khi xử lý xong, ghi log kết quả (Status code)
    # Không cần dùng BackgroundTasks ở đây vì middleware chạy độc lập
    write_system_log(user_name, method, path, response.status_code, ip)
    
    return response

Trong FastAPI, middleware được xếp chồng như lớp vỏ hành. Lệnh app.add_middleware khai báo sau sẽ nằm ở lớp bên trong.
1. Khi Request đến: Nó đi qua log_requests (lớp ngoài) -> rồi mới qua SessionMiddleware (lớp trong).
2. Khi Response quay về: Nó đi từ SessionMiddleware (đã nạp session vào request) -> rồi mới quay lại log_requests. Lúc này, hàm log_requests mới có thể đọc được request.session.
đưa SessionMiddleware xuống sau @app.middleware("http")

Trong kiến trúc FastAPI hiện đại (hoặc các dự án lớn), Controller đóng vai trò là "người điều phối" (Orchestrator).
Dưới đây là cách phân chia nhiệm vụ mới:
Router (routers/): Định nghĩa URL (ví dụ: /departments/save).
Controller (controllers/): Nhận Request, gọi Service phù hợp, xử lý lỗi HTTP (400, 404, 500) và trả về JSON.
Service (services/): Chứa logic nghiệp vụ (kiểm tra trùng tên, kiểm tra nhân viên).
CRUD (crud/): Chỉ chứa các câu lệnh SQL thuần túy (SELECT, INSERT, UPDATE).
Tại sao Router "càng lười" thì dự án càng tốt?
Tránh lặp code: Nếu bạn viết logic trả về {"status": "success"} ở Router, khi bạn có 100 cái Router, bạn phải viết 100 lần. Nếu sau này sếp muốn đổi thành {"result": "ok"}, bạn phải đi sửa 100 chỗ.
Tách biệt trách nhiệm: Router lo việc "Đường đi", Controller lo việc "Phản hồi", Service lo việc "Tính toán".
Dễ bảo trì: Khi bạn sửa cấu trúc JSON trong Controller, mọi Router gọi nó sẽ tự động cập nhật theo mà không cần chạm vào file Router.
@router.post("/api/save")
async def save_department(data: DepartmentSaveRequest, db: Connection = Depends(get_db)):
    return DepartmentController.save(db, data)
